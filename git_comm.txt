файл .gitignore - игнорирование файлов
   Символ (*) соответствует 0 или более символам
    [abc] — любому символу из указанных в скобках
    [0-9] — соответствуют любому символу из интервала
    a/**/z - использовать две звёздочки, чтобы указать на вложенные каталоги: a/**/z соответствует a/z, a/b/z, a/b/c/z
    # Исключить все файлы с расширением .a
    *.a
    # Но отслеживать файл lib.a даже если он подпадает под исключение выше
    !lib.a
    # Исключить файл TODO в корневом каталоге, но не файл в subdir/TODO
    /TODO
    # Игнорировать все файлы в каталоге build/
    build/
    # Игнорировать файл doc/notes.txt, но не файл doc/server/arch.txt
    doc/*.txt
    # Игнорировать все .txt файлы в каталоге doc/
    doc/**/*.txt

git diff - непосредственно добавленные и удалённые строки — патч как он есть(только те, что ещё не проиндексированы)
git diff --staged(git diff --cached) -Выведет все изменения, сделанные в рабочей директории которые были добавлены в индекс


$ git commit - зафиксировать изменения    p 
git commit -m "комментарий к изменению"
git commit -a - автоматически индексировать каждый уже отслеживаемый на момент коммита файл, позволяя вам обойтись без git add

rm PEOPLE.md - удаление файла из рабочей дерриктории
# Любое изменение нужно добавлять в индекс
git add PEOPLE.md
git commit -m 'remove PEOPLE.md'
После изменений всегда делается git add, который подготавливает изменение к коммиту (а не добавляет файл). После этого выполняется коммит.

git rm объединяет в себе удаление и подготовку к коммиту (Равносильно rm + git add)

git restore <имя удаленного файла> - восстановление удаленного файла

git log - список всех коммитов, сверху последние
git log -p - коммиты с полным дифом кнопка f, промотать назад — b, выйти из режима просмотра — q

git show <первые 7мь символов хэша коммита(всегда уникальные)> - можно посмотреть все изменения, сделанные в рамках одного коммита

git blame <путь до файла> - выводит файл и рядом с каждой строчкой показывает того, кто ее менял и в каком коммите

git grep ищет совпадение с указанной строкой во всех файлах проекта(что изменение строчки — не то же самое, что ее написание. Вполне возможно, что программист исправил небольшую опечатку, а саму строчку написал кто-то до него) Имея такой вывод, уже легко пойти дальше и изучить конкретный коммит.

git grep -i позволяет искать без учета регистра
git grep <искомое слово> 5120bea - поиск в конкретном коммите
# Поиск по всей истории
# Возвращаем список хешей коммитов `rev-list`
git grep <искомое слово> $(git rev-list --all)

git log - перечисляет коммиты, сделанные в репозитории в обратном к хронологическому порядке
git log -p(--patch) - показывает разницу (выводит патч), внесённую в каждый коммит -2 -ограничить количество записей в выводе команды
git log --stat увидеть сокращённую статистику для каждого коммита, печатает под каждым из коммитов список и количество изменённых файлов, а также сколько строк в каждом          из файлов было добавлено и удалено

git commit --amend  команда использует область подготовки (индекс) для внесения правок в коммит. Если вы ничего не меняли с момента последнего коммита (например, команда запущена сразу после предыдущего коммита), то снимок состояния останется в точности таким же, а всё что вы сможете изменить — это ваше сообщение к коммиту. В итоге получится единый коммит — второй коммит заменит результаты первого. 
Когда вы вносите правки в последний коммит, вы не столько исправляете его, сколько заменяете новым, который полностью его перезаписывает. В результате всё выглядит так, будто первоначальный коммит никогда не существовал, а так же он больше не появится в истории вашего репозитория.
git reset HEAD <file> для исключения из индекса(вы изменили два файла и хотите добавить их в разные коммиты)
git checkout -- <file> если вы поняли, что не хотите сохранять свои изменения файла



 Работа с удаленными репозиториями

git remote - выведет названия доступных удалённых репозиториев
	-v - вывод с адресами
	
git remote add <shortname> <url> - добавление нового удаленного репозитория
Можно обращаться через <shortname> git fetch pb

$ git fetch [remote-name] - забирает данные в ваш локальный репозиторий, но не сливает их с какими-либо вашими наработками и не модифицирует то, над чем вы работаете в данный момент. Вам необходимо вручную слить эти данные с вашими, когда вы будете готовы.

git pull - если ветка настроена на отслеживание удалённой ветки, автоматически получить изменения из удалённой ветки и слить их со своей текущей

git remote show - показывает какая именно локальная ветка будет отправлена на удалённый сервер по умолчанию при выполнении git push, также показывает, каких веток с удалённого сервера у вас ещё нет, какие ветки всё ещё есть у вас, но уже удалены на сервере, какие удалённые ветки будут в них влиты при выполнении git pull. 

git remote show <remote> - выдаёт URL удалённого репозитория, а также информацию об отслеживаемых ветках

git remote rename - для переименования удалённого репозитория

git remote rm - удалить удалённый репозиторий, ссылки на удалённый репозиторий все отслеживаемые ветки и настройки, связанные с этим репозиторием, так же будут удалены
